Autowagen Master — Credit notes & customer balances

How returns and credits work, how the invoice, Accounts receivable report, and customer statement stay aligned. Save as PDF: Ctrl+P → Save as PDF (Chrome / Edge).

File location: open this page from your project at docs/credit_notes_system_guide_print.html in a browser (double-click or drag into Chrome).

1. What a credit note is (in this system)

A credit note is a formal reduction linked to an original final sales invoice (number like INV-YYYY-NNNNN). After you finalize the credit note, the system assigns a number like CN-YYYY-NNNNN.

You never “invent” a standalone credit in the POS screen — you start from a final invoice and create a credit note that points at that invoice. That keeps the audit trail clear for you and for your customer.

2. Three rules you asked for (product design)

  1. Link to the original invoice. Every credit note row in the database stores invoice_id and lines reference sales_invoice_lines. The UI shows the linked INV-… and has a button to open that invoice.
  2. Stock is restored for part lines when the credit note is finalized. For each credit line that has a part_id, the system increases qty_on_hand by the returned quantity (not forced to 1). If the part was sold and stock becomes positive again, status can move back to available.
  3. AR reduction vs cash refund. On the credit note you choose one adjustment type:
    • AR reduction — means “reduce what the buyer still owes on this invoice relationship” from an accounts-receivable point of view; you fill credit date and notes.
    • Cash refund — means money was returned to the buyer; you must record refund paid date, method (cash / EFT / card / other), and optionally a reference. That metadata is stored on the credit note for your paper trail.

3. How “remaining balance” is calculated (one formula)

After finalized credit notes exist, every screen uses the same idea:

Amount still due on the invoice = Invoice total (incl. VAT) − Sum of finalized credit notes for that invoice − Sum of active payments on that invoice
  • Invoice screen (POS) — under Payments, you see credits (if any), paid total, and Remaining. New payments cannot exceed that remaining amount.
  • Accounts receivable report — customer and invoice rows use the same net idea so “who owes us” matches what you see on each invoice.
  • Customer statement — each invoice shows Total, Paid, Credits, and Balance; the balance column is the same net as above.

Why this stays consistent: all three places read from the same underlying facts: final invoice totals, finalized credit totals from sales_credit_notes, and payment lines from sales_invoice_payments. There is no separate “Excel side calculation” inside the app for AR.

5. Draft vs finalized credit notes

  • Draft — you can change return quantities, dates, and adjustment type; nothing hits stock yet; totals on reports usually count only final credits.
  • Finalize — assigns CN-…, restores stock for part lines, and locks the note. Caps apply: you cannot finalize a credit total that pushes all credits above the invoice face value.

6. Staff: where to click

  1. Run SQL file sql/07_credit_notes.sql once in phpMyAdmin if your database does not yet have credit tables.
  2. Open a final invoice → New credit note (or Reports → Credit notes → create from invoice).
  3. Set return quantities, adjustment type, save draft, then finalize when correct.
  4. Check the same invoice’s Payments header: credits and remaining should update.
  5. Open Reports → Accounts receivable and the customer’s Statement — balances should match the invoice logic above.

7. Optional future rule (not the default today)

You may later decide that only “AR reduction” credits should reduce receivables metrics, while cash refund credits are tracked for compliance but treated differently in “who owes us” maths (for example when the buyer already paid in full and you refund cash).

Today’s build intentionally applies all finalized credits to the remaining balance everywhere, so invoice / AR / statement stay simple and aligned. Changing that is a deliberate product decision and would need code + training updates.

8. Showing your shop to buyers (advertising) — viewer vs public website

A. Prefer the public shop (best for advertising)

For customers browsing parts you should send them to the public pages (no login):

  • Parts shop: /shop/ on your server (example: http://localhost/autowagen-master/shop/ on your PC, or your live domain in production).
  • Stripping / wreck listings: /shop/stripping/

Those URLs are meant for browsers; you don’t expose your full back-office POS or master data.

B. “Viewer” login (inside the office only)

The database allows a role viewer. Viewer users sign in through the normal login page but many buttons are hidden or read-only (lists, dashboards) — intended for demos or trusted observers, not for random internet traffic.

To create a viewer user (owner does this once):

  1. Open Laragon → Menu → Terminal (or any terminal where php works).
  2. Run a one-off command to generate a password hash (replace the password inside the quotes with a strong temporary password):
    php -r "echo password_hash('YourStrongPasswordHere', PASSWORD_DEFAULT), PHP_EOL;"
  3. Copy the long hash line.
  4. Open phpMyAdmin → database autowagen_master → table users → Insert.
  5. Fill username, full_name, paste into password_hash, set role = viewer, is_active = 1, save.
  6. Go to your site auth/login.php and sign in with that username and password.

Safety: do not publish viewer credentials online. For marketing, use /shop/. Use viewer only behind your own supervision or VPN if you truly need someone to see the back office without editing.

9. Note for developers (PHP syntax check)

Some PCs do not have the php program on the system PATH. That only affects running commands like php -l file.php from a generic terminal. Laragon still ships PHP — use Laragon’s terminal or the full path to php.exe under C:\laragon\bin\php\…. It does not change how the website runs in Apache.
